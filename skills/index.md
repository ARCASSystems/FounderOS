The setup wizard (`founder-os-setup`) is the entry point. All others activate via natural language phrasing or via `/founder-os:<command>`.

**Reading the `Runs on:` line.** Every skill's `SKILL.md` declares its runtime class on a `Runs on:` line near the top: `reasoning` (read and reason; any capable agent), `local-writes` (create or edit OS files; needs folder write access, so Claude Code or a folder-attached surface like Cowork or Antigravity, while a web-only surface drafts the change for you), or `local-exec` (run a local script; needs a local-runtime agent like Claude Code, while a cloud surface reads the produced artifacts instead). The class is the highest capability the skill's happy path needs. The full contract and the honest-degradation rule live in `CLAUDE.md`.

### Role packs

The skills are organised into role packs. A pack is a function a solo founder covers alone, opened by one front-door wedge skill that routes into the members. It is a naming convention plus a manifest, not a folder: every skill stays a top-level directory and is available regardless of pack. The front door does more than you ask - you name the outcome, it aims the right member at it and invites you into the rest without forcing.

| Pack | Front door | Manifest | Role it covers |
| --- | --- | --- | --- |
| LinkedIn / Audience | [linkedin-start](linkedin-start/SKILL.md) | [linkedin-pack.md](linkedin-pack.md) | CMO / BD |
| Pipeline | [pipeline-start](pipeline-start/SKILL.md) | [pipeline-pack.md](pipeline-pack.md) | BD |
| Content | [content-start](content-start/SKILL.md) | [content-pack.md](content-pack.md) | CMO |
| Delivery | [delivery-start](delivery-start/SKILL.md) | [delivery-pack.md](delivery-pack.md) | COO |
| Money | [unit-economics](unit-economics/SKILL.md) | [money-pack.md](money-pack.md) | CFO |
| Decisions | [decisions-start](decisions-start/SKILL.md) | [decisions-pack.md](decisions-pack.md) | Chief of Staff |

The brain and meta skills (audit, verify, brain-pass, query, lint, wiki-build, knowledge-capture, memory-pass, cross-link, the save / history / restore / backup verbs, routines) and the voice scaffold (add-voice) stay as shared substrate, not sold as role-packs. They are the engine the packs run on.

### LinkedIn Pack

A logical pack (not a folder): every member is named `linkedin-*` and shares one front door and one algorithm reference. You bring one file - your LinkedIn data export - and pick an outcome; the OS aims the same data there. Members: `linkedin-start` (the front door), `linkedin-network-scan` (leads and jobs), `linkedin-power-audit` (brand and network-health), `linkedin-brand-direction` (content direction), `linkedin-warm-revival` (dormant contacts), and `linkedin-post` (content execution). The shared reference is `skills/linkedin-pack-references/linkedin-algorithm.md`. Full manifest: `skills/linkedin-pack.md`. Everything runs locally, free-plan, within LinkedIn's terms.
