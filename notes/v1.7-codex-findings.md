## WS1, WS2, WS3 build notes

- **WS1 templates side-effect.** `templates/scripts/query.py` was already present in the v1.6 codebase but had not been re-mirrored after WS1's `top_entry_id` / ID-output additions. WS2 closed that gap by re-mirroring `templates/scripts/query.py` byte-for-byte against `scripts/query.py`. Worth noting for future workstreams that touch `scripts/query.py`: the templates mirror needs the same patch, and there is no automated check enforcing parity.

- **WS1 example IDs in `entry-conventions.md`.** The "Examples" subsection in `templates/rules/entry-conventions.md` keeps realistic-digit IDs as illustrative content. `query.py`'s `top_entry_id` will pick these up if a user installs `entry-conventions.md` into their working `rules/` directory and `rules/` is in the scan scope. The trade-off (concrete examples vs false-positive in query results) was made in favour of concrete examples. If the false-positive surfaces in real use, future fix path is either (a) replace illustrative IDs with placeholders, or (b) scope `top_entry_id` to skip files in `rules/` and `templates/`.

- **WS3 dual-hook scenario on Windows.** Both `post-tool-use-observation.sh` and `post-tool-use-observation.ps1` are registered. The bash hook now detects MSYS/MINGW/Cygwin and exits 0 silently, so on Windows the PowerShell hook is the authoritative writer. Documented in `templates/brain/observations/README.md` under "Platform behavior."

- **WS3 PowerShell BOM fix.** `Add-Content -Encoding UTF8` on Windows PowerShell 5.1 writes a BOM on first file create. Replaced with `[System.IO.File]::AppendAllText(..., new UTF8Encoding(false))` which is BOM-free on both 5.1 and 7+.

- **WS3 manual-JSON-escape hardening.** The fallback path (no jq, no python) now strips control chars (`\000-\037\177`) before escaping. `INTENT` strip and PowerShell intent filter were both updated to also drop DEL (0x7F).
